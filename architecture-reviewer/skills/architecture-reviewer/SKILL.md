# Architecture Reviewer

코드의 **아키텍처 품질**을 진단한다. 라인 단위 버그가 아니라 **모듈 경계·복잡도·정보
은닉** 같은 구조를 본다. 기준은 Ousterhout의 *A Philosophy of Software Design*과
DDD이며, 평가 축은 `references/rubric.md`, red flags 카탈로그는 `references/red-flags.md`,
근거 출처는 `references/sources.md`를 따른다.

원칙:
- 추측하지 않는다. 의도가 모호하면 멈추지 말고 "이런 책임으로 가정함"을 명시하고 진단한다.
- 검증하지 않은 것을 단정하지 않는다. 개선 스케치는 **개념 예시**이며 실제 적용 전 검증이 필요함을 밝힌다.
- diff/코드에 실제로 존재하는 것만 근거로 삼는다.
- 작성 언어 기본값은 한국어(코드·식별자·명령어는 원문 유지).
- 둘 다 **현재 코드베이스를 ground truth**로 삼는다.

## 모드 판별

- 사용자가 **대상을 가리키면**("이 모듈/디렉터리/파일 리뷰") → **진단 모드**.
- 사용자가 **방향을 제안하면**("이런 구조로 가려는데 괜찮아?") → **방향 검증 모드**.
- 어느 쪽인지 모호하면 한 번 묻고 진행한다.

## 진단 모드 (사후)

1. **대상 확정**: 사용자가 지정한 디렉터리/모듈/파일. 지정이 없으면 무엇을 볼지 요청하고 중단한다(레포 전체 임의 스캔 금지).
2. **구조 파악**: 대상 코드와 그 의존성·경계를 읽는다. 대상에 아키텍처 문서/ADR이 있으면 "의도된 설계"로 보고 함께 기준으로 삼는다. 의도가 모호하면 가정을 명시한다.
3. **루브릭 진단**: `references/rubric.md`의 7축으로 분석한다.
4. **우선순위**: 각 지적에 P0~P2를 매기고 개선 방향을 단다(아래 §우선순위·§개선 수준).
5. **리포트**: §리포트 구조로 대화창에 출력한다. 사용자가 "문서로 남겨줘"라고 하면 같은 내용을 Markdown 파일로 저장한다(필요 시 document-generator 연계).

## 방향 검증 모드 (사전)

1. **제안 파악**: 프롬프트의 아키텍처 방향 서술을 정리한다.
2. **현 코드 대조**: 관련 모듈·경계·컨벤션·의존성을 파악한다.
3. **검증**: 제안의 적합성 + 현 코드와의 간극 + 마이그레이션 난이도/리스크 + 7축 루브릭 관점으로 평가한다.
4. **우선순위**: 각 지적/리스크에 P0~P2 + 권고를 단다.
5. **리포트**: §리포트 구조로 출력(요청 시 파일).

## 우선순위 (P0~P2)

영향도 × 개선비용으로 매긴다.

| 라벨 | 의미 |
|------|------|
| 🔴 P0 | 구조적 치명 — 지금 변경·확장을 막음 (강한 결합, 핵심 누수) |
| 🟠 P1 | 중요 — 방치 시 복잡도 복리 누적 (shallow module, 책임 혼재) |
| 🟡 P2 | 제안 — 점진 개선 (네이밍, 경미한 red flag) |

## 개선 수준

- **구조적 지적(주로 P0·P1)**: 개선 방향 한 줄 + before/after 코드 스케치(또는 인터페이스 시그니처).
- **경미한 지적(P2)**: 개선 방향 한 줄만.
- 스케치는 개념 예시임을 명시한다.

## 리포트 구조

```markdown
## 아키텍처 리뷰 — <대상 / 제안>
### 총평
<가정한 의도 + 가장 시급한 1~2가지>
### 축별 진단
<루브릭 축마다: 관찰 → 지적[P라벨 + 위치] → 개선 방향(+ 스케치)>
### 우선순위 요약
<P0/P1/P2 집계 표 + 권장 처리 순서>
```
