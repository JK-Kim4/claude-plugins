# Document Generator

Claude Code로 작업하다 보면 결과를 보고하거나, 진행을 추적하거나, 설계를 남기거나, 사용법을 안내하는 문서를 자주 만들게 된다. 이때 목적과 독자에 맞지 않는 형식으로 쓰면 정작 필요한 정보가 빠지거나 군더더기가 생긴다. 이 스킬은 **프롬프트에서 문서의 목적·독자·유형을 먼저 판단하고, 그에 맞는 검증된 형식을 골라 채우는 것**이 핵심이다.

## 동작 원칙

좋은 문서는 "예쁜 템플릿"에서 나오지 않고 **"누가, 왜 읽는가"에 맞춘 정보 선택**에서 나온다. 형식은 그 판단을 돕는 출발점일 뿐이다. 그래서 이 스킬은 항상 같은 순서로 움직인다: 목적 판단 → (모호하면) 짧게 질문 → 형식 선택 → 실제 내용 수집 → 작성.

## 1단계: 문서의 목적과 유형 판단

사용자의 프롬프트에서 다음을 먼저 읽어낸다.

- **독자가 누구인가** — 나중의 나/팀원/리뷰어/처음 보는 사람? 독자가 아는 것이 적을수록 배경 설명을 더 담는다.
- **무엇을 위해 읽는가** — 무슨 일이 됐는지 알려고? 다음에 뭘 할지 추적하려고? 왜 이렇게 설계했는지 이해하려고? 어떻게 쓰는지 따라하려고?
- **출력 형식** — 사용자가 형식(파일 확장자, 위치, 특정 포맷)을 명시했으면 그대로 따른다. 명시가 없으면 **Markdown 파일**이 기본이다.
- **언어** — 기본은 **한국어**. (코드·식별자·명령어·고유명사는 원문 유지.) 사용자가 다른 언어를 요청하면 따른다.

판단 결과를 아래 유형 중 하나로 매핑한다. 각 유형의 상세 템플릿은 `references/`에 있고, 모두 신뢰도 높은 1차 출처(Google eng-practices, Conventional Commits, Keep a Changelog, Nygard/MADR ADR, AWS, Diátaxis, Write the Docs 등)에 근거해 작성됐다. 유형이 정해지면 **해당 파일을 읽고 그 구조와 권고를 따른다.** 각 파일 하단의 출처는 권고의 근거이지 문서에 인용할 대상이 아니다.

| 유형 | 언제 | 핵심 질문 | 템플릿 |
|------|------|-----------|--------|
| **작업 보고서** | 완료한 작업을 요약·보고. "정리해줘", "리포트", "PR 설명", "변경 요약" | 무엇을, 왜 했고, 결과가 무엇인가 | `references/work-report.md` |
| **진행/추적** | 진행 상황을 추적·관리. "체크리스트", "TODO", "진행 현황", "마일스톤" | 무엇이 끝났고 무엇이 남았나 | `references/progress-tracking.md` |
| **기술 설계/분석** | 설계 결정·분석을 기록. "설계 문서", "ADR", "기술 분석", "트러블슈팅 정리" | 왜 이 결정을 했고 대안은 무엇이었나 | `references/technical-design.md` |
| **온보딩/가이드** | 사용법·절차를 안내. "README", "가이드", "셋업 안내", "사용법" | 독자가 따라서 성공할 수 있나 | `references/onboarding-guide.md` |

유형이 둘 이상에 걸치면 **주된 목적 하나**를 골라 그 형식을 뼈대로 삼고, 필요한 섹션만 다른 유형에서 빌려온다. 억지로 한 문서에 모든 유형을 욱여넣지 않는다.

## 2단계: 모호하면 짧게 질문

프롬프트만으로 목적·독자·범위가 충분히 잡히지 않을 때는 **추측하지 말고 1~2개의 핵심 질문**을 한 뒤 진행한다. 매번 묻는 게 아니라, 답에 따라 문서 구조가 실제로 달라질 때만 묻는다. 좋은 질문의 예:

- "이 보고서는 팀 공유용인가요, 본인 기록용인가요?" (배경 설명 분량이 갈림)
- "설계 문서인가요, 아니면 이미 한 작업의 보고인가요?" (유형이 갈림)
- "추적 문서를 새로 만들까요, 기존 파일에 이어 쓸까요?" (위치가 갈림)

답이 명확하면 가정을 한 줄로 밝히고("팀 공유용 보고서로 작성합니다") 바로 진행한다.

## 3단계: 실제 내용 수집

문서의 가치는 형식이 아니라 **정확한 내용**에서 나온다. 빈칸을 추측으로 메우지 말고 출처에서 사실을 확인한다.

- 방금 한 작업이면 **대화 기록(transcript)**에서 무엇을 바꿨는지, 왜 그랬는지, 무엇이 검증됐는지 끌어온다.
- 코드 변경이면 `git diff`, `git log`, 변경된 파일을 직접 읽어 사실을 확인한다.
- 확인되지 않은 결과(테스트 통과 여부 등)는 단정하지 않는다. 모르면 "미확인"으로 표시하거나 확인한다.

## 4단계: 작성

선택한 템플릿 구조에 수집한 내용을 채운다. 작성 시 공통 원칙:

- **요약을 맨 위에.** 바쁜 독자가 첫 문단/표만 보고도 핵심을 알 수 있어야 한다.
- **추측과 사실을 구분한다.** 검증된 것, 추정인 것, 미확인인 것을 섞지 않는다.
- **빈 섹션은 넣지 않는다.** 템플릿의 섹션이라도 해당 내용이 없으면 뺀다. 빈 헤더는 노이즈다.
- **파일로 저장한다.** 사용자가 위치를 지정하지 않았으면 프로젝트 루트나 `docs/`에 의미 있는 이름(`kebab-case.md`)으로 저장하고, 저장 경로를 알려준다.
- 분량은 목적에 맞게. 추적 문서는 짧고 빽빽하게, 설계 문서는 맥락을 충분히.

**출력 형식 분기:** 사용자가 **HTML**(또는 스타일·디자인이 입혀진 문서, PDF)을 요청하면 `references/html-output.md`를 읽고 그 규칙을 따른다. 그 가이드는 내용·구조는 위 종류별 템플릿을 그대로 따르되, 문서 종류에 맞는 시각적 컴포넌트(RAG 상태 헤더, ADR status 배지, Breaking Change 경고 박스, 단계별 기대결과 박스 등)를 입혀 외부 의존성 없는 자기완결 HTML 파일로 출력하는 방법을 담고 있다. 별도 형식 요청이 없으면 Markdown이 기본이다.

**Notion 출력 분기:** 사용자가 **Notion**에 만들어 달라고 하면(예: "노션에 정리해줘", "노션 페이지로 만들어줘") `references/notion-output.md`를 읽고 따른다. 내용·구조는 종류별 템플릿을 그대로 따르되 표현을 Notion 블록(callout/columns/to-do/table 등)으로 입히고, `notion-create-pages`로 페이지를 만든다. 저장 위치(parent)는 모호하면 짧게 확인한다.

**다이어그램/시각화:** 구조·흐름·상태·데이터 관계를 그림으로 보여줘야 하면 채널에 맞는 방식을 쓴다. **HTML이면 인라인 SVG**(`references/diagrams.md` — 외부 의존 없이 자기완결, 그리드 좌표 규칙으로 어긋남 방지), **Notion이면 mermaid 코드블록**(`references/notion-output.md` — Notion이 네이티브 렌더). 어느 쪽이든 ASCII 아트는 쓰지 않는다. 기술 설계 문서에서 특히 유용하다.

작성 후, 어떤 유형으로 판단했고 어디에 저장했는지 한 줄로 요약해 사용자에게 알린다.
