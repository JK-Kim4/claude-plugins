# 코드 리뷰 기준 (축 B)

pr-reviewer가 코드 리뷰를 할 때 참고하는 체크리스트다. diff에 실제로 존재하는
변경만 대상으로 하고, 불확실한 지적은 "가능성"으로 명시한다.

## 1. 보안 취약점

- **인젝션**: SQL/명령/LDAP 등 사용자 입력이 검증·파라미터화 없이 쿼리·셸로 전달되는가.
- **비밀정보 노출**: API 키·토큰·비밀번호 하드코딩, 로그에 민감정보 출력.
- **인증·인가 누락**: 권한 체크 없이 보호 자원 접근, 권한 상승 가능 경로.
- **안전하지 않은 역직렬화**: 신뢰 불가 입력을 객체로 역직렬화.
- **경로·SSRF**: 사용자 입력으로 파일 경로·URL을 구성(path traversal, SSRF).
- **암호화**: 약한 해시/암호 알고리즘, 검증 없는 랜덤.

## 2. 동시성 이슈 가능 지점

- **공유 상태 경쟁**: 동기화 없이 여러 스레드/고루틴이 가변 상태를 읽고 쓴다.
- **락**: 락 누락, 락 순서 불일치(데드락 가능), 락 범위 과대/과소.
- **비원자적 연산**: check-then-act, read-modify-write 가 원자적이지 않다.
- **스레드 안전성**: 스레드 안전하지 않은 컬렉션·객체를 공유.
- **비동기**: await 누락, 미처리 promise/future, 취소·타임아웃 부재.

## 3. 로직 오류

- **경계 조건**: off-by-one, 빈 컬렉션·단일 요소, 최대/최소값.
- **null/예외 처리**: null 역참조 가능, 빈 catch로 예외 삼킴, 자원 누수(미해제).
- **분기**: 잘못된 비교 연산자, 누락된 else/default, 중복·도달 불가 분기.
- **계산**: 정수 오버플로·나눗셈 0, 부동소수 비교, 단위·부호 실수.

## 4. 우선순위(P0~P2) 매핑

| 라벨 | 의미 | 대표 예 |
|------|------|---------|
| 🔴 P0 | 치명 — 머지 전 반드시 수정 | 명확한 보안 취약점, 데이터 손상·크래시 유발 로직 오류 |
| 🟠 P1 | 중요 — 수정 권장 | 동시성 이슈, 버그 가능성, 팀 강제 컨벤션 위반 |
| 🟡 P2 | 제안 — 선택적 개선 | 가독성, 경미한 컨벤션·스타일 |

판정 지침: 확실히 악용·오작동하면 P0, 조건부로 문제될 수 있으면 P1, 동작에 영향 없는 개선은 P2.
